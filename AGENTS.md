- 与用户交流必须使用中文。
- 代码注释必须使用中文。
- 文档（README、设计文档、计划文档）优先使用中文；如需英文术语，可中英文并列。

## 项目定位

这是一个基于 **Cocos Creator 3.8.6** 的斯诺克游戏项目，目标平台是：

- Web
- Mobile Web
- 原生移动端（如后续接入 Android / iOS 打包）

- 架构清晰，便于逐步扩展
- 输入系统同时兼容鼠标与触摸
- UI 与桌台表现能够适配横屏移动设备和网页
- 核心玩法逻辑与表现层解耦


## 开发原则

### 1. 逻辑与表现分离

斯诺克项目后续会很容易变复杂，因此必须尽量避免把规则、物理、UI、动画写在同一个脚本里。


### 2. 优先兼容 Web 与 Touch

这是一个要同时跑在网页和移动设备上的项目，所以所有输入方案都应该默认考虑：

- Mouse
- Touch
- 高 DPI 屏幕
- 横屏布局
- 屏幕比例变化

不要把交互绑定成只能鼠标操作。所有瞄准、拖动力度、镜头或视图调整，都要设计成触摸也能自然完成。

### 3. 优先做“稳定可玩”，再追求“复杂物理”

斯诺克体验对物理非常敏感，但在早期开发阶段，不要一开始就过度追求绝对真实。建议优先保证：

- 击球方向稳定
- 球移动稳定
- 球停下判定稳定
- 入袋判定稳定
- 回合切换稳定

在这些稳定之后，再逐步微调：

- friction
- restitution
- spin
- cushion bounce
- pocket behavior

## 斯诺克核心系统建议

### 1. 球体系统

至少要明确以下数据：

- `id`
- `ballType`
- `scoreValue`
- `isPotted`
- `velocity`
- `angularVelocity`
- `position`

建议将“红球 / 彩球 / 白球”的差异尽量数据化，不要写大量硬编码分支。

### 2. 击球系统

击球系统建议拆成三部分：

- `Aim`：瞄准方向
- `Power`：力度
- `Strike`：实际施加给母球的 impulse

如果后续要支持 English、spin、低杆、高杆，也建议继续在这个系统里扩展，而不是散落到 UI 或球体脚本里。

### 3. 回合与规则系统

斯诺克规则比普通台球复杂，建议单独维护一个 `RuleSystem` 或 `MatchSystem`，处理：

- 当前应击打目标球
- 是否 first contact 合法
- 是否犯规
- 本杆得分
- 彩球是否需要 respot
- 一杆结束条件
- 全部球清台后的流程

不要把规则判定直接写进 `BallController`。

### 4. 停球检测

“所有球是否静止”会直接影响：

- 是否允许下一次击球
- 是否结算犯规
- 是否切换回合
- 是否刷新 UI

因此必须有统一的停球检测逻辑。不要由多个地方分别判断“看起来差不多停了”。

## 跨端适配建议

### 1. 屏幕方向

这个项目更适合 **landscape**。后续开发默认以横屏作为主体验。

### 2. 分辨率适配

建议基于 `Canvas + Widget + UITransform` 做自适应，并注意：

- 超宽屏设备
- 平板比例
- 手机刘海 / 安全区
- 浏览器窗口变化

关键 UI 不要贴边过紧。

### 3. 输入抽象

建议统一封装一个输入层，例如：

- `PointerInput`
- `GestureInput`

上层只消费统一事件，不直接区分鼠标还是触摸。这样 Web 与 Mobile 才不会写出两套逻辑。

## 性能建议

面向 Web 与移动端时，优先注意：

- 减少频繁创建和销毁节点
- 使用 object pooling 管理可复用对象
- 避免在 `update()` 中写过重逻辑
- 把高频计算集中到必要模块
- 音频、贴图、特效资源分级加载

如果后续要引入更复杂的物理或特效，先在中低端手机上验证帧率表现。

## 后续开发优先级建议
1. 建立规范目录结构
2. 完成基础场景与桌台布局
3. 完成母球与目标球的数据模型
4. 完成基础击球与球体移动
5. 完成停球检测
6. 完成入袋判定
7. 完成基础规则与回合切换
8. 完成 UI 与跨端交互优化
9. 最后再调手感、特效、音频和高级规则细节

## 协作约定

如果后续由人类开发者或 AI agent 一起参与这个项目，建议统一遵守：

- 修改前先确认 Creator 版本仍为 `3.8.x`
- 优先新增脚本，不随意改动场景序列化内容
- 修改 `.scene`、`.prefab` 时避免无关重存，减少 diff 噪音
- 新增系统时先定义职责边界，再开始写代码
- 优先保持可运行、可验证的小步提交
- 不要为了兼容旧版脚本接口而保留冗余适配层；如果接口设计已经不合理，可以直接改成更适合当前版本的结构。
- 不要为了兼容历史关卡数据、旧字段名或旧节点绑定而扭曲新实现；必要时直接更新数据、场景绑定和调用方。
- 完成新功能开发后同步更新AGENTS.md文件，确保使用最新的内容。

## 当前实现状态（2026-03-22）

### 已落地的第一版结构

- `assets/scripts/config/SnookerConfig.ts`
  - 统一维护设计分辨率、桌台尺寸、物理参数、练习布局和 10 个示例关卡。
- `assets/scripts/core/SnookerTypes.ts`
  - 维护球类型、模式、阶段、球状态和关卡配置类型。
- `assets/scripts/core/PointerInput.ts`
  - 统一封装鼠标与触摸输入，供上层消费统一事件。
- `assets/scripts/core/SnookerPhysics.ts`
  - 使用轻量自定义二维物理，负责摩擦、碰撞、落袋、停球检测。
- `assets/scripts/core/SnookerRules.ts`
  - 负责本杆得分、白球犯规、BREAK 结算和文案输出。
- `assets/scripts/ui/UiFactory.ts`
  - 负责程序化 UI 节点、圆角面板、圆形节点和文本节点构建。
- `assets/scripts/game/SnookerGame.ts`
  - 作为游戏主入口，负责主界面、游戏 HUD、结算界面、比赛流程和桌台表现。
- `assets/scenes/Main.scene`
  - 第一版入口场景，默认挂载 `SnookerGame`。

- 已实现：

### 后续协作补充

- 如果继续扩展，请优先保持“输入 / 规则 / 物理 / UI”职责分离。
