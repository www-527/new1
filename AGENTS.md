# AGENTS.md

## 1. 语言与文档

- 与用户交流必须使用中文。
- 代码注释必须使用中文。
- 文档（README、设计文档、计划文档）优先使用中文；如需保留英文术语，可采用中英文并列。

## 2. 项目概览

这是一个基于 **Cocos Creator 3.8.6** 的斯诺克游戏项目，当前目标平台包括：

- Web
- Mobile Web
- 原生移动端（后续可接入 Android / iOS 打包）

项目整体目标：

- 架构清晰，便于逐步扩展
- 输入系统同时兼容鼠标与触摸
- UI 与桌台表现适配横屏移动设备和网页
- 核心玩法逻辑与表现层解耦

## 3. 核心开发原则

### 3.1 逻辑与表现分离

斯诺克项目后续复杂度会持续上升，必须尽量避免把规则、物理、UI、动画写进同一个脚本。

### 3.2 优先兼容 Web 与 Touch

所有输入交互默认同时考虑以下场景：

- Mouse
- Touch
- 高 DPI 屏幕
- 横屏布局
- 屏幕比例变化

不要把交互设计成只能鼠标操作。所有瞄准、拖动力度、镜头或视图调整，都要保证触摸操作也自然可用。

### 3.3 优先做稳定可玩，再追求复杂物理

开发早期先保证以下基础体验稳定：

- 击球方向稳定
- 球移动稳定
- 球停下判定稳定
- 入袋判定稳定
- 回合切换稳定

在上述基础稳定后，再逐步微调：

- friction
- restitution
- spin
- cushion bounce
- pocket behavior

## 4. 核心系统职责建议

### 4.1 球体系统

至少明确以下数据字段：

- `id`
- `ballType`
- `scoreValue`
- `isPotted`
- `velocity`
- `angularVelocity`
- `position`

红球、彩球、白球的差异应尽量数据化，不要依赖大量硬编码分支。

### 4.2 击球系统

击球系统建议拆分为三层：

- `Aim`：瞄准方向
- `Power`：力度
- `Strike`：对母球施加实际 impulse

若后续支持 English、spin、低杆、高杆，也应继续扩展在这一层内，不要分散到 UI 或球体脚本。

### 4.3 回合与规则系统

建议单独维护 `RuleSystem` 或 `MatchSystem`，负责：

- 当前应击打目标球
- 是否 `first contact` 合法
- 是否犯规
- 本杆得分
- 彩球是否需要 respot
- 一杆结束条件
- 全部球清台后的流程

不要把规则判定直接写进 `BallController`。

### 4.4 停球检测

“所有球是否静止”会直接影响：

- 是否允许下一次击球
- 是否结算犯规
- 是否切换回合
- 是否刷新 UI

必须有统一的停球检测逻辑，不要由多个位置各自判断“是否差不多停了”。

## 5. 跨端适配与性能

### 5.1 屏幕与布局

- 项目默认以 **landscape** 横屏体验为主。
- 建议基于 `Canvas + Widget + UITransform` 做分辨率适配。
- 需要重点关注：
  - 超宽屏设备
  - 平板比例
  - 手机刘海与安全区
  - 浏览器窗口变化

关键 UI 不要贴边过紧。

### 5.2 输入抽象

建议统一封装输入层，例如：

- `PointerInput`
- `GestureInput`

上层只消费统一事件，不直接区分鼠标或触摸，避免 Web 与 Mobile 维护两套交互逻辑。

### 5.3 性能约束

面向 Web 与移动端时，优先注意：

- 减少频繁创建和销毁节点
- 使用 object pooling 管理可复用对象
- 避免在 `update()` 中堆积过重逻辑
- 把高频计算集中到必要模块
- 音频、贴图、特效资源分级加载

若后续引入更复杂的物理或特效，先在中低端手机上验证帧率表现。

## 6. 建议开发顺序

1. 建立规范目录结构
2. 完成基础场景与桌台布局
3. 完成母球与目标球的数据模型
4. 完成基础击球与球体移动
5. 完成停球检测
6. 完成入袋判定
7. 完成基础规则与回合切换
8. 完成 UI 与跨端交互优化
9. 最后再调手感、特效、音频和高级规则细节

## 7. 协作约定

如果后续由人类开发者或 AI agent 协作开发，统一遵守以下规则。

### 7.1 Creator 与序列化文件

- 修改前先确认 Creator 版本仍为 `3.8.x`。
- 优先新增脚本，不随意改动场景序列化内容。
- 修改 `.scene`、`.prefab` 时避免无关重存，减少 diff 噪音。
- 新增系统时先定义职责边界，再开始写代码。
- 优先保持可运行、可验证的小步提交。

### 7.2 资源文件与 `.meta`

- 如果新增了图片、音频、Prefab、场景或其他 Creator 资源文件，AI agent 不要手写、猜测或补造对应 `.meta`。
- 遇到新增资源时，应明确告知用户打开 Cocos Creator 编辑器导入或刷新资源，并由编辑器生成 `.meta` 文件。

### 7.3 资源引用方式

- 图片等资源优先直接挂到场景节点、Prefab 节点或已序列化组件引用上使用。
- 尽量不要在运行时通过路径做动态加载。
- 如果确实需要动态创建或按需生成内容，优先通过 `Prefab` 引用和实例化实现。

### 7.4 接口与历史包袱

- 不要为了兼容旧版脚本接口而保留冗余适配层；如果接口设计已经不合理，可以直接改成更适合当前版本的结构。
- 不要为了兼容历史关卡数据、旧字段名或旧节点绑定而扭曲新实现；必要时直接更新数据、场景绑定和调用方。

### 7.5 文档同步

- 完成新功能开发后同步更新 `AGENTS.md`，确保协作约定和当前实现状态保持最新。

## 8. 当前实现状态（2026-03-24）

### 8.1 已落地的第一版结构

- `assets/scripts/config/SnookerConfig.ts`
  - 统一维护设计分辨率、桌台尺寸、物理参数与对局配置；当前同时保留休闲模式与专家模式两套开局配置，前者用于短局体验，后者使用标准斯诺克开局布局，包含 15 颗红球、6 颗彩球与母球的标准站位。
- `assets/scripts/core/SnookerTypes.ts`
  - 维护球类型、模式、阶段、球状态和配置类型；当前对外主流程以模式选择入口驱动，后续扩展仍可复用同一套阶段与状态定义。
- `assets/scripts/core/PointerInput.ts`
  - 统一封装鼠标与触摸输入，供上层消费统一事件；已补充编辑器预览态下的节点失效保护，避免组件销毁或热重载时因空节点解绑事件而在控制台报错。
- `assets/scripts/core/SnookerPhysics.ts`
  - 使用轻量自定义二维物理，负责摩擦、碰撞、落袋、停球检测。
- `assets/scripts/core/SnookerRules.ts`
  - 负责本杆得分、白球犯规、BREAK 结算和文案输出；已修复对局内结算提示的乱码问题，当前击球反馈与犯规提示会输出正常中文。
- `assets/scripts/ui/UiFactory.ts`
  - 负责场景节点复用与程序化 UI 补齐，统一处理圆角面板、圆形节点和文本节点的创建/复用。
- `assets/scripts/ui/SnookerTheme.ts`
  - 维护参考设计图提炼出的视觉主题色盘与材质倾向，供主界面、HUD、桌台和按钮统一复用。
- `assets/scripts/ui/TextureSkinner.ts`
  - 当前用于把资源目录中的面板、按钮、桌框、台呢和袋口贴图接到运行时关键节点，确保程序化布局也能复用正式 PNG 资源；若后续切回全静态场景绑定，应同步清理这层运行时贴图入口。
- `assets/scripts/game/SnookerGame.ts`
  - 作为游戏主入口，当前首屏已切换为模式选择 UI，提供休闲模式与专家模式两种入口；对局内三段式 HUD、底部蓄力区与战报浮层继续由运行时组织，`TouchLayer` 已调整到 HUD 下方以避免拦截按钮点击；顶部 HUD 保留暂停图标按钮，重开本局仅放在弹层中；首杆开球会先高亮 D 区并隐藏白球，玩家需先在 D 区点击放置白球，再从白球位置拖拽瞄准开球；暂停、结算等通用弹层会根据当前按钮数量自动重排位置，避免双按钮场景出现错层或不对齐；重开本局时会先把旧球节点移出 `BallLayer` 再销毁，避免同帧重建时复用待销毁节点导致首按清屏；按钮层已增加触摸/鼠标事件去重，辅助线开关会同步刷新按钮皮肤，并在瞄准中立即重绘预测线，球体、瞄准线、碰撞预测、袋口高光和浮字反馈继续保持运行时创建；对局内上方战报浮层、力度档位和球色提示文案已改为稳定的 Unicode 文本，避免编辑器或脚本编码异常导致显示问号。
- `assets/scenes/Main.scene`
  - 当前作为精简入口场景使用，默认仅保证 `GameRoot` 正确挂载 `SnookerGame`；启动时会先清空 `GameRoot` 下历史静态子节点，再由脚本重建模式选择页、HUD、桌台和结算层，避免旧序列化布局干扰新版运行时视觉。
- `assets/resources/textures/`
  - 存放首批正式 PNG 贴图资源，包含背景、面板、按钮、台呢、木框和袋口纹理。
- `assets/resources/prefabs/ui/`
  - 保留 `CommonButton`、`TopHud`、`BottomHud` 三个 UI 预制体作为可复用资产参考；当前主流程仍以 `SnookerGame.ts` 运行时组织布局为主，这些 prefab 暂作为后续回切静态场景方案时的资源参考。
- `doc/main.png`
  - 当前模式选择首页的视觉参考图，作为最新 UI 布局与材质风格的对照基准。
- `tools/generate_arcade_textures.py`
  - 根据当前街机台球厅风格生成正式贴图资源；后续若要换风格或微调材质，应优先修改该脚本后批量重生贴图。
- `tools/generate_ui_prefabs.py`
  - 根据当前图片资源与已有 `.meta` 生成 UI 预制体，并补齐 `sprite-frame` 子资源；如果图片已在 Creator 中导入，应保留现有 uuid 再重生 prefab。
- `tools/attach_scene_ui_sprites.py`
  - 批量把 `Main.scene` 中的静态面板、桌台、袋口和按钮节点挂上 `cc.Sprite` 及正式贴图引用；当前主流程不直接依赖这些静态节点，但如果后续回切为场景直绑方案，优先重跑该脚本再整理序列化层级。
